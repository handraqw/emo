# Максимально подробный гайд по дополнительному обучению системы

Этот документ описывает, как дообучать и постепенно заменять baseline-эвристики проекта на более сильные обученные модели.

---

## 1. Что именно можно обучать в проекте

В текущем состоянии проект состоит из нескольких логических модулей, и почти каждый можно усилить отдельным обучением:

1. **Face detection** — детекция лица.
2. **Face emotion recognition** — эмоция по лицу.
3. **Speech-to-text** — распознавание речи.
4. **Voice emotion recognition** — эмоция по голосу.
5. **Text toxicity / text emotion** — токсичность, сарказм, речевые паттерны.
6. **Fusion layer** — объединение модальностей.

Важно: эти модули не обязательно обучать одновременно. На практике лучше двигаться поэтапно.

---

## 2. Правильная стратегия развития

Рекомендуемый порядок такой:

### Этап 1. Стабилизировать данные

Сначала нужно сделать качественный датасет:

- видео;
- аудио;
- выравнивание по времени;
- аннотации эмоций;
- контроль качества разметки.

### Этап 2. Обучить сильную модель по лицу

Это обычно даёт самый понятный и быстрый выигрыш.

### Этап 3. Обучить voice emotion

Голос сильно помогает в случаях, где лицо нейтрально, частично закрыто или находится вне кадра.

### Этап 4. Подключить хороший STT + текстовую модель

Текст даёт важный сигнал для сарказма, грубости, угроз и конфликтов.

### Этап 5. Обучить fusion

Когда отдельные модальности стали достаточно сильными, можно переходить от ручных весов к обучаемому fusion-блоку.

---

## 3. Сбор данных: что нужно хранить

Минимальная единица данных для хорошего мультимодального обучения:

```text
sample/
 ├── video.mp4
 ├── audio.wav                # можно извлекать автоматически
 ├── transcript.txt           # опционально, если есть ручная расшифровка
 ├── face_segments.csv        # кадры/отрезки с лицом
 ├── emotion_labels.csv       # эмоции по времени
 └── metadata.json            # служебные поля
```

### 3.1. Что желательно аннотировать

- `start_ms`, `end_ms`
- лицо присутствует / не присутствует
- говорящий один / несколько
- основная эмоция
- вторичная эмоция
- уверенность разметчика
- токсичность / сарказм / грубость
- шум, качество света, качество микрофона

### 3.2. Почему нужны интервалы, а не только whole-video label

Потому что эмоция почти всегда меняется во времени.  
Если размечать только всё видео целиком, система плохо учится:

- находить момент переключения эмоции;
- синхронизировать лицо, голос и текст;
- объяснять конфликт модальностей.

---

## 4. Как подготовить разметку

### 4.1. Формат таблицы аннотаций

Удобный CSV-вариант:

```csv
video_id,start_ms,end_ms,face_emotion,voice_emotion,text_emotion,final_emotion,transcript
clip_001,0,1800,NEUTRAL,IRRITATION,NEUTRAL,IRRITATION,"ладно, давай быстрее"
clip_001,1800,3200,ANGER,AGGRESSION,RUDE,AGGRESSION,"ты вообще слушаешь или нет"
```

### 4.2. Правила для разметчиков

Нужно заранее зафиксировать:

- различие `ANGER` и `AGGRESSION`;
- различие `SARCASM` и `RUDE`;
- когда ставить `NEUTRAL`;
- как размечать смешанные эмоции;
- что делать, если лицо и голос противоречат друг другу.

### 4.3. Проверка межэкспертного согласия

Желательно иметь минимум двух разметчиков на часть датасета и считать:

- Cohen’s kappa;
- percent agreement;
- disagreement matrix.

Если kappa низкая, проблема чаще всего не в модели, а в плохих инструкциях разметки.

## 5. Дообучение face emotion

Файл, который уже существует: `src/face_model/train_face_emotion.py`

Сейчас он baseline-уровня, но именно его проще всего расширять.

### 5.1. Что подавать на вход

Оптимальный input:

- crop лица;
- выровненное изображение;
- 224x224;
- нормализованные каналы.

### 5.2. Что можно использовать как backbone

- ResNet18
- EfficientNet-B0
- MobileNetV3
- ViT-tiny / small

Для MVP часто лучший компромисс:

- **ResNet18** для простоты;
- **MobileNetV3** для скорости.

### 5.3. Loss function

Если одна эмоция на sample:

```text
CrossEntropyLoss
```

Если эмоций несколько:

```text
BCEWithLogitsLoss
```

### 5.4. Полезные техники

- class weights для дисбаланса;
- focal loss, если мало редких эмоций;
- label smoothing;
- strong augmentations, но без разрушения выражения лица.

### 5.5. Аугментации

Можно:

- brightness / contrast;
- mild blur;
- resize / crop;
- compression artifacts;
- небольшие повороты.

Нельзя злоупотреблять:

- сильными affine-деформациями;
- очень агрессивным color jitter;
- cutout по ключевым зонам лица.

### 5.6. Что сохранять после обучения

Минимум:

- веса модели;
- список классов;
- preprocessing config;
- метрики на validation;
- confusion matrix.

## 6. Дообучение speech-to-text

Сейчас проект ожидает, что Vosk-модель лежит локально и используется через `SpeechToTextService`.

### 6.1. Два реальных пути

#### Вариант A. Использовать готовую Vosk-модель

Плюсы:

- быстро;
- локально;
- мало интеграционных изменений.

Минусы:

- ограниченное качество;
- сложнее адаптировать под домен.

#### Вариант B. Перейти на современную STT-модель

Например:

- Whisper / faster-whisper;
- NeMo;
- wav2vec2 / HuBERT / Conformer.

### 6.2. Что нужно для дообучения STT

- пары `audio -> transcript`;
- чистые тексты;
- нормализация цифр / пунктуации;
- доменные слова и имена.

### 6.3. Главные метрики

- **WER** — word error rate
- **CER** — character error rate

Формула WER:

```text
WER = (S + D + I) / N
```

где:

- `S` — substitutions
- `D` — deletions
- `I` — insertions
- `N` — число слов в эталоне

Чем меньше WER, тем лучше.

### 6.4. Почему STT влияет на эмоции

Если распознавание речи плохое:

- текстовая токсичность ломается;
- сарказм теряется;
- fusion становится шумным;
- `CONFLICT` определяется хуже.

## 7. Дообучение voice emotion recognition

Это один из самых важных направлений для данного проекта.

### 7.1. Что подавать в модель

Обычно используются:

- log-Mel spectrogram;
- MFCC;
- prosodic features;
- raw waveform (реже для простого MVP).

### 7.2. На каких данных учить

Нужны короткие speech segments:

- 1–4 секунды;
- один говорящий;
- хорошая синхронизация;
- эмоция размечена по интервалу.

### 7.3. Какие модели подходят

- CNN по spectrogram;
- CRNN;
- wav2vec2 + classifier head;
- ECAPA-TDNN + emotion head;
- AST / HTS-AT для audio transformers.

### 7.4. Что считать baseline-feature extractor

Даже если остаётся rule-based UI, полезно считать:

- `pitch mean/std`
- `energy mean/std`
- speaking rate
- pause ratio
- jitter / shimmer

### 7.5. Loss

Как правило:

```text
CrossEntropyLoss
```

Иногда полезно:

- focal loss;
- metric learning;
- multi-task loss.

### 7.6. Почему стоит делать multi-task

Например одновременно предсказывать:

- emotion;
- arousal;
- valence.

Это стабилизирует обучение и делает признаки более переносимыми.

## 8. Дообучение текстовой части

Сейчас `src/text_toxicity.py` — rule-based модуль.

Его можно заменить на:

- ruBERT classifier;
- DeBERTa/RoBERTa для русского;
- multi-label classifier для `AGGRESSION/RUDE/SARCASM`.

### 8.1. Входные данные

- расшифрованный текст;
- ручные transcript-аннотации;
- текст из субтитров;
- при наличии — диалоговый контекст.

### 8.2. Почему контекст важен

Сарказм часто не определяется по одной фразе.  
Нужны:

- предыдущие реплики;
- интонация;
- выражение лица;
- ситуация до и после.

### 8.3. Режим multi-label

Хорошая практика — учить не одну метку, а несколько:

- toxicity
- sarcasm
- insult
- threat
- calm / neutral

Тогда fusion получает более богатый сигнал.

## 9. Как обучать fusion

Сейчас `fusion_engine.py` использует фиксированные веса:

```text
face = 0.40
voice = 0.35
text = 0.25
```

Это хороший baseline, но не оптимум.

### 9.1. Что можно сделать лучше

Собрать признаки:

- вероятности face model;
- вероятности voice model;
- вероятности text model;
- confidence detector’ов;
- флаг отсутствия лица;
- качество аудио;
- длина реплики;
- наличие конфликта модальностей.

И обучить сверху:

- logistic regression;
- XGBoost / LightGBM;
- MLP;
- attention-based fusion.

### 9.2. Почему сначала не надо усложнять fusion

Если отдельные модальности слабые, сложный fusion не спасёт.  
Сначала нужно поднять качество каждого канала отдельно.

## 10. Как строить датасет для fusion

Для каждой временной точки или сегмента нужно сохранить:

```text
segment_id
face_probs[...]
voice_probs[...]
text_probs[...]
face_present
audio_quality
transcript_length
gold_final_emotion
```

Это позволит обучить fusion как обычную supervised classification задачу.

## 11. Практический пайплайн обучения по шагам

### Шаг 1. Соберите сырые данные

- реальные видео;
- камера / вебка / записанные ролики;
- разные уровни шума;
- разные лица, ракурсы, освещение.

### Шаг 2. Нарежьте данные

- по видео-сценам;
- по speech segments;
- по face tracks.

### Шаг 3. Сделайте train/val/test split

Важно делать split по людям и сессиям, а не по кадрам.  
Иначе будет leakage.

### Шаг 4. Подготовьте feature pipelines

- image transforms;
- audio transforms;
- transcript cleaning;
- label mapping.

### Шаг 5. Обучите по отдельности

1. face model
2. voice model
3. text model

### Шаг 6. Замерьте отдельно каждую модальность

Без этого невозможно понять, где узкое место.

### Шаг 7. Соберите fusion dataset

Прогоните train/val/test через три модели и сохраните их выходы.

### Шаг 8. Обучите fusion

Сначала простую logistic regression / MLP.

### Шаг 9. Проверьте real-world cases

Нужны сценарии:

- лицо есть, аудио нет;
- лицо нет, аудио есть;
- текст пустой;
- конфликт между улыбкой и токсичной речью;
- сарказм при нейтральном лице;
- шумная улица;
- плохой свет.

## 12. Как оценивать качество

### 12.1. Для лица

- accuracy
- macro F1
- per-class recall
- confusion matrix

### 12.2. Для голоса

- macro F1
- weighted F1
- robustness under noise

### 12.3. Для текста

- F1 по toxicity/sarcasm labels
- precision on aggressive text

### 12.4. Для STT

- WER
- CER

### 12.5. Для fusion

- macro F1 на final emotion
- отдельно precision/recall для `CONFLICT`

### 12.6. Почему macro F1 важнее accuracy

Потому что классы эмоций обычно дисбалансны, и accuracy легко завышается за счёт `NEUTRAL`.

Формула F1:

```text
F1 = 2 * (precision * recall) / (precision + recall)
```

---

## 13. Как избежать leakage и ложного роста метрик

Самые частые ошибки:

1. один и тот же человек попал и в train, и в test;
2. соседние фрагменты одного видео оказались в разных split;
3. нормализация делается по всему датасету сразу;
4. гиперпараметры подогнаны под test.

Правильный подход:

- group split по speaker / session / video;
- отдельный validation;
- test использовать только для финальной оценки.

- `src/speech_to_text.py`


- `src/fusion_engine.py`


## 18. Итог

Проект уже построен так, что baseline-модули можно по одному заменять на обученные модели без полного переписывания приложения.  
Самое важное для успешного дополнительного обучения — не “сразу обучить всё”, а:

1. сначала выстроить качественные данные;
2. потом усилить отдельные модальности;
3. только после этого учить fusion;
4. обязательно проверять реальные кейсы, а не только offline-метрики.
